## 1. Understanding the Execution Plan

The SQL Server Query Optimizer produces a **cost‑based execution plan** that describes how it will retrieve the requested data. Think of the plan as a recipe – if the optimizer chooses the wrong ingredients, your query will run slowly.

### 1.1 Graphical vs. XML

- **Graphical plans** (available in SSMS/Azure Data Studio) are easier to read and show operators as icons.
- **Actual plans** include runtime statistics (rows, CPU, I/O) and are generated with `SET STATISTICS XML ON` or by checking "Include Actual Execution Plan".

```sql
SET STATISTICS XML ON;
SELECT * FROM dbo.Orders WHERE OrderID = 123;
SET STATISTICS XML OFF;
```

Use the `sys.dm_exec_query_plan(plan_handle)` DMV to retrieve plans programmatically.

### 1.2 Common plan red flags

- **Table/Index scans** on large tables when a seek is expected.
- **Nested loops** joining millions of rows – switch to hash merge or sort when appropriate.
- **Remote queries** or blocking operations.
- **Missing index suggestions** in the plan tooltip; capture them and evaluate.

### 1.3 Cost vs. Actual Rows

Remember the optimizer's cost estimates can be wildly off if statistics are stale. Compare *Estimated Subtree Cost* to *Actual Rows* and make choices accordingly.

```sql
SELECT
    qs.total_logical_reads,
    qs.execution_count,
    qp.query_plan
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
WHERE qs.total_logical_reads > 100000;
```

This query surfaces heavy-read plans so you can focus tuning efforts.

## 2. Indexing Strategies

Indexes are the foundation of query performance. Poor indexing is the single largest cause of slow queries.

### 2.1 Index types & when to use them

| Index Type | Use case | Notes |
|------------|----------|-------|
| Clustered | Primary key, determines physical order | One per table |
| Nonclustered covering | Reporting queries | Include non-key columns |
| Filtered | Sparse data (e.g., active customers) | Smaller & faster |
| Columnstore | Analytical workloads, data warehouse | Great for aggregations |
| XML/SP/CLR | Specialized queries | Use sparingly |

### 2.2 Creating covering and filtered indexes

Covering indexes eliminate lookups by including required columns:

```sql
CREATE NONCLUSTERED INDEX IX_Orders_CustomerDate
ON dbo.Orders(CustomerID, OrderDate)
INCLUDE (TotalAmount, Status);
```

Filtered index example for active users:

```sql
CREATE NONCLUSTERED INDEX IX_Users_Active
ON dbo.Users(IsActive)
WHERE IsActive = 1;
```

### 2.3 Columnstore & batch mode

```sql
CREATE CLUSTERED COLUMNSTORE INDEX CCI_Sales ON dbo.Sales;
```

Columnstore indexes enable **batch mode** processing, cutting CPU for OLAP queries.

### 2.4 Maintenance and fragmentation

Weekly or nightly jobs should:

```sql
ALTER INDEX ALL ON dbo.Orders REORGANIZE;
ALTER INDEX ALL ON dbo.Orders REBUILD WITH (FILLFACTOR = 90);
```

Use `sys.dm_db_index_physical_stats` to monitor fragmentation.

## 3. Query Refactoring and SARGability

A query is *SArgable* when the optimizer can use indexes instead of scanning rows.

### 3.1 Predicate rewrites

Bad:
```sql
SELECT * FROM Orders
WHERE YEAR(OrderDate) = 2025;
```
Good:
```sql
SELECT * FROM Orders
WHERE OrderDate >= '2025-01-01'
  AND OrderDate < '2026-01-01';
```

Avoid calculations on indexed columns, use `BETWEEN` or range predicates.

### 3.2 Set-based operations

Replace cursors and row-by-row logic with set operations or `MERGE`:

```sql
UPDATE o
SET o.Status = 'Delayed'
FROM dbo.Orders o
JOIN dbo.Delays d ON o.OrderID = d.OrderID;
```

### 3.3 Temporary structures strategically

Sometimes a well–placed temp table or table variable reduces repeated expensive subqueries:

```sql
SELECT CustomerID, SUM(Amount) AS Total
INTO #CustomerTotals
FROM dbo.Payments
GROUP BY CustomerID;

SELECT * FROM #CustomerTotals WHERE Total > 100000;
```

### 3.4 Query hints & plan guides

Use `OPTION(FORCESEEK)` or create a **plan guide** when the optimizer repeatedly chooses a bad plan.

```sql
SELECT * FROM dbo.Products
WHERE ProductName LIKE 'A%'
OPTION (FORCESEEK);
```

## 4. Statistics & Parameter Sniffing

Statistics inform the optimizer about data distribution. Bad stats = bad plans.

### 4.1 Keeping stats up-to-date


```sql
EXEC sp_updatestats; -- updates auto-created stats
```

For critical tables, schedule full scans:

```sql
UPDATE STATISTICS dbo.Orders WITH FULLSCAN;
```

### 4.2 Auto options

Enable:
```sql
ALTER DATABASE CURRENT SET AUTO_CREATE_STATISTICS ON;
ALTER DATABASE CURRENT SET AUTO_UPDATE_STATISTICS ON;
ALTER DATABASE CURRENT SET AUTO_UPDATE_STATISTICS_ASYNC ON;
```

Async updates prevent queries from waiting for stats to refresh.

### 4.3 Parameter sniffing issues

SQL Server caches a plan using the first parameter values it sees. If those values are atypical, subsequent executions may suffer.

Problem query:
```sql
CREATE PROCEDURE GetOrders @CustID INT
AS
SELECT * FROM Orders WHERE CustomerID = @CustID;
```

Fixes:
- Use `OPTION (RECOMPILE)` inside the proc.
- Use `DECLARE @cid INT = @CustID;` to trick the optimizer.
- Add a plan guide forcing a generic plan.

## 5. Hardware & Configuration Tips

Software tuning can't overcome poorly configured hardware.

### 5.1 Memory

Set `max server memory` to leave room for the OS. Monitor `sys.dm_os_sys_memory`.

### 5.2 Tempdb

- One data file per CPU core (up to 8) to avoid allocation contention.
- Place on SSD/NVMe; avoid network storage.

### 5.3 Disk I/O

Separate your data, log, and tempdb on different arrays. Use **RAID 10** for latency-sensitive workloads.

### 5.4 Parallelism

Adjust `MAXDOP` per workload:
```sql
EXEC sp_configure 'max degree of parallelism', 4;
RECONFIGURE;
```

OLTP often benefits from 1‑4; OLAP may use all cores.

### 5.5 NUMA awareness

For servers with NUMA, ensure SQL Server is configured to take advantage of node-local memory and CPUs.

## 6. Monitoring and Tools

Tracking what’s happening in production is essential for long‑term performance.

### 6.1 Dynamic Management Views (DMVs)

Queries worth running regularly:

```sql
-- Top CPU consuming queries
SELECT TOP 20
    qs.total_worker_time/qs.execution_count AS avg_cpu,
    qt.text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
ORDER BY avg_cpu DESC;
```

### 6.2 Extended Events

Create a lightweight session that captures long-running queries:

```sql
CREATE EVENT SESSION long_queries ON SERVER
ADD EVENT sqlserver.sql_statement_completed(
    ACTION(sqlserver.sql_text)
    WHERE duration > 1000000)
ADD TARGET package0.event_file(SET filename='C:\xe\long_queries.xel');
ALTER EVENT SESSION long_queries ON SERVER STATE=START;
```

### 6.3 Query Store

Enable Query Store to compare plans over time and force a good plan:

```sql
ALTER DATABASE CURRENT SET QUERY_STORE = ON;
ALTER DATABASE CURRENT SET QUERY_STORE (OPERATION_MODE = READ_WRITE);
```

Then use the GUI or DMVs like `sys.query_store_runtime_stats`.

## 7. Real‑World Example: Financial Application

A mid‑size bank was experiencing 500 ms latency on a daily transaction query that ran millions of times per day. After a full review:

1. Added a covering index that included a computed column.
2. Rewrote a scalar UDF as an inline TVF because UDFs serialized the plan.
3. Identified a parameter sniffing issue and added `OPTION (RECOMPILE)` to the stored procedure.
4. Adjusted max memory and moved tempdb to NVMe drives with eight data files.
5. Enabled Query Store and forced the optimal plan for the problem query.

**Results:**
- Average latency dropped from 500 ms to 40 ms.
- CPU utilization decreased by 65% during peak hours.
- The system could handle twice the expected transaction load without scaling hardware.
